Containerization for LLMs

Containerization is the foundation of modern cloud-native applications, especially for AI/ML workloads. Containers provide isolated, reproducible environments that ensure your LLM applications run consistently across different environments, from development to production.

Docker Containers
Lightweight, portable containers that package your LLM application and all its dependencies into a single unit.
Kubernetes Orchestration
Automated deployment, scaling, and management of containerized LLM applications across clusters of machines.

Docker Fundamentals

Docker is the de facto standard for containerization. It allows you to package your LLM application with all its dependencies into a lightweight, portable container that can run anywhere Docker is installed.

Dockerfile for LLM Application:

# Multi-stage build for LLM application FROM python:3.10-slim as base # Install system dependencies RUN apt-get update && apt-get install -y \\ gcc \\ g++ \\ && rm -rf /var/lib/apt/lists/* # Set working directory WORKDIR /app # Copy requirements and install Python dependencies COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # Production stage FROM base as production # Copy application code COPY . . # Create non-root user for security RUN useradd --create-home appuser \\ && chown -R appuser:appuser /app USER appuser # Expose port EXPOSE 8000 # Health check HEALTHCHECK --interval=30s --timeout=30s --start-period=5s --retries=3 \\ CMD curl -f http://localhost:8000/health || exit 1 # Start the application CMD ["python", "app.py"]

LLM-Specific Considerations

Containerizing LLM applications requires special considerations due to their resource requirements, model storage needs, and performance characteristics.

Memory Management
LLMs require significant memory. Use resource limits and requests to ensure proper allocation and prevent OOM kills.
Model Storage
Large model files need persistent storage. Use volumes and consider model caching strategies for performance.

Optimized Dockerfile for LLM:

# Multi-stage build for LLM with GPU support
FROM nvidia/cuda:12.1.0-runtime-ubuntu22.04 as base

# Install Python and system dependencies
RUN apt-get update && apt-get install -y \\
    python3.10 \\
    python3-pip \\
    && rm -rf /var/lib/apt/lists/*

# Install NVIDIA runtime
RUN apt-get update && apt-get install -y \\
    nvidia-container-toolkit \\
    && rm -rf /var/lib/apt/lists/*

FROM base as builder

# Install Python dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir torch torchvision torchaudio \\
    && pip install --no-cache-dir -r requirements.txt

FROM base

# Copy Python binaries from builder
COPY --from=builder /usr/local/lib/python3.10/site-packages /usr/local/lib/python3.10/site-packages
COPY --from=builder /usr/local/bin /usr/local/bin

# Copy application code
COPY . /app
WORKDIR /app

# Create model directory
RUN mkdir -p /models

# Set environment variables
ENV CUDA_VISIBLE_DEVICES=0
ENV PYTORCH_CUDA_ALLOC_CONF=max_split_size_mb:512

# Health check for LLM service
HEALTHCHECK --interval=30s --timeout=30s --start-period=5s --retries=3 \\
    CMD python -c "import torch; print('GPU available:', torch.cuda.is_available())" || exit 1

CMD ["python", "llm_service.py"]

Service Deployment

Deploying LLM services requires careful consideration of networking, load balancing, and service discovery to ensure high availability and performance.

Load Balancing
Distribute incoming requests across multiple LLM service instances to handle high traffic and ensure availability.
Persistent Storage
Store model files, configuration, and user data using persistent volumes and storage classes.

Ingress Configuration for LLM Service:

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: llm-ingress
  annotations:
    kubernetes.io/ingress.class: nginx
    cert-manager.io/cluster-issuer: letsencrypt-prod
    nginx.ingress.kubernetes.io/rate-limit: "100"
    nginx.ingress.kubernetes.io/rate-limit-window: "1m"
    nginx.ingress.kubernetes.io/proxy-body-size: "10m"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "300"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "300"
spec:
  tls:

    - llm.example.com
    secretName: llm-tls-secret
  rules:
  - host: llm.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: llm-service
            port:
              number: 80

Scaling Strategies

LLM workloads can vary significantly in demand. Implementing proper scaling strategies ensures your application can handle peak loads while optimizing costs during low usage periods.

Horizontal Scaling
Add more instances of your LLM service to handle increased load. Kubernetes can automatically scale based on CPU/memory usage.
Vertical Scaling
Increase resources (CPU, memory, GPU) for individual instances to handle more complex or larger requests.

Horizontal Pod Autoscaler (HPA):

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: llm-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: llm-service
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 10
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60
      - type: Pods
        value: 2
        periodSeconds: 60
      selectPolicy: Min
